1. Architecture Overview

ValueSkins has three layers that never mix responsibilities:

Frontend

What users see. Enforces nothing. Displays only. All decisions come from the server.

Backend

Where all logic, security, data, and enforcement live. 24 Rust microservices behind a single API gateway.

Database

PostgreSQL. 30 versioned migrations. Append-only. Every state change is auditable.

Infrastructure

Stateless services. Horizontal scaling. Feature flags. Canary deployment compatible.

The Core Rule

The client (browser or phone) never makes decisions. The frontend disables buttons for UX. The backend enforces the same rules independently. A developer calling the API directly hits the same gates as the UI. There is no way to bypass enforcement by writing custom requests.

2. Frontend — Next.js 16 + React 19 + TypeScript

What is React?

React is a JavaScript library for building user interfaces, invented at Facebook in 2013. Instead of rewriting the entire page when data changes, React only updates the specific parts that changed. This makes it fast and efficient. Version 19 (what we use) includes the React Compiler — automatic performance optimization without manual code changes.

Meta relevance: React was built by Facebook/Meta. Pitching a React frontend to Meta is like pitching a Ford engine to Ford. Their engineers know it deeply. Integration into Instagram's existing React codebase is straightforward.

What is Next.js?

Next.js is a production framework built on top of React, made by Vercel. It adds: file-based routing (each file in /app becomes a URL), server-side rendering (pages pre-built on the server for SEO and speed), API routes, automatic code splitting, and zero-config deployment. Version 16 is the latest stable release.

What is TypeScript?

TypeScript is JavaScript with mandatory type declarations. If you write a function that expects a number and someone passes a string, TypeScript refuses to compile. This eliminates entire categories of runtime bugs before they reach production. The entire frontend has zero TypeScript errors — verified by running npx tsc --noEmit after every change.

3. Backend — Rust + Actix-Web + Tokio

What is Rust?

Rust is a systems programming language developed by Mozilla Research, first released in 2015. It compiles to native machine code — the same level as C and C++. What makes it different: memory safety is enforced at compile time through a concept called ownership. Every piece of data has exactly one owner. When the owner goes out of scope, the memory is freed automatically — no garbage collector needed. The compiler prevents you from accessing freed memory, creating data races between threads, or having null pointer dereferences. If code compiles in Rust, it is memory-safe by proof.

Meta relevance: Meta contributed to Rust's development and uses it in production infrastructure. Cloudflare, Discord, AWS, Dropbox, and the Linux kernel all use Rust for their most performance-critical components. Saying "we use Rust" to Meta's CTO signals serious engineering intent.

What is Actix-Web?

Actix-Web is a Rust web framework for building HTTP APIs. In TechEmpower benchmarks — the industry standard for measuring web framework performance — Actix-Web consistently ranks as the fastest web framework in existence. It handles hundreds of thousands of concurrent HTTP connections per second on a single server. It is what we use to expose all 24 microservice APIs.

What is Tokio?

Tokio is Rust's async runtime. When a request is waiting for a database response (which takes milliseconds), Tokio uses that thread to process other requests instead of blocking. This is how a single server handles millions of simultaneous users efficiently. Think of it as a highly optimized task scheduler built into the language runtime.

4. Microservices — 24 Independent Services

What are microservices?

Instead of one giant application that handles everything, the backend is split into small, focused services. Each service does one thing and does it well. They communicate through the API gateway. If you need to update the matching logic, you deploy only the matching service — auth, payments, and notifications are untouched and unaffected.

Restaurant analogy: A kitchen with separate stations — grill, salad, dessert, drinks. If the dessert station is slow, the grill still operates at full speed. A monolith is one chef doing everything — one bottleneck, one failure point, everything stops.

5. Database — PostgreSQL + SQLx + 30 Migrations

What is PostgreSQL?

PostgreSQL is the most advanced open-source relational database, first released in 1996 and actively maintained. It handles complex queries with joins across multiple tables, JSON data, concurrent writes with full ACID guarantees, and scales to billions of rows. Instagram uses PostgreSQL. Meta uses it. It is the industry standard for production social platforms.

What is SQLx?

SQLx is a Rust library for database access. Unlike ORMs (Object-Relational Mappers) that hide SQL behind abstraction layers and generate inefficient queries, SQLx lets you write real SQL that is verified at compile time. If you write a query that references a column that doesn't exist in the schema, it fails to compile — not at runtime when a real user hits it. This eliminates an entire category of production database errors.

The 30 migrations — what they are

A migration is a SQL script that makes a specific, versioned change to the database schema. They are numbered sequentially and run in order. This means every environment — development, staging, production — has the exact same schema. The history of every schema change is tracked in version control exactly like code.

Key tables: users, personas, professions, persona_professions (the ValueSkin assignment), opportunities (campaigns), applications, deal_rooms, offer_rounds, fraud_rules (admin-configurable thresholds), discovery_rate_limits (sliding window), reputation_seeds (cold-start reputation), onboarding_deals, account_creation_log

What makes it production ready?

Indexes on hot columns: Every foreign key, every status column, every created_at column has a database index. Queries stay fast as data grows from 1,000 to 1 billion rows. Without indexes, a query that takes 1ms at 1,000 rows takes 10 minutes at 1 billion rows.

Parameterized queries: All database queries use bound parameters — WHERE id = $1 with the value passed separately. SQL injection is structurally impossible. Not "we sanitize inputs" — the SQL and data are separate at the protocol level.

Connection pooling: 20 connections, 5 minimum, 5-second acquire timeout, 5-minute idle timeout, 30-minute max lifetime. The pool is managed by the shared library used across all 24 services.

No unbounded queries: Every list endpoint has LIMIT and OFFSET. Hard cap at 100 results per page. A query that works in 1ms at 10 rows causes a database outage at 10 million rows without limits.

Append-only design: Data is never destructively deleted — it is marked inactive. Every state change is an insert, not an update to a previous record. Full audit trail of every action.

Q: Why not NoSQL (MongoDB, DynamoDB)?
ValueSkins has deeply relational data: a persona belongs to a user, has professions, has applications, has deal rooms, has offers, has reputation scores. Relational data belongs in a relational database. NoSQL would make joins across these entities significantly harder and slower, requiring multiple round-trips to the database for data that PostgreSQL returns in a single query. Instagram itself uses PostgreSQL for exactly this reason.
Q: How does it scale beyond one database server?
PostgreSQL supports read replicas natively — write operations go to the primary, read operations (the vast majority of traffic) go to replicas. For ValueSkins, the matching queries are read-heavy and go to replicas while writes go to primary. This is standard PostgreSQL scaling and requires zero application code changes — only configuration. Meta already operates PostgreSQL at this scale for Instagram.
Q: What if you need to change the schema after launch?
Write migration 31. It runs sequentially after migration 30. Because we use append-only design, adding columns is safe and non-breaking. Removing columns requires a multi-step process: deprecate in code, deploy, then remove column. The migration system enforces discipline around schema changes.

6. Authentication & Security

Instagram OAuth 2.0 — How login actually works

Step 1: User clicks "Login with Instagram" on frontend

Step 2: Browser redirects to Instagram's authorization page (instagram.com, not our server)

Step 3: User approves — Instagram redirects back to /auth/callback?code=ABC123&state=XYZ

Step 4: Our backend receives the one-time code. Exchanges it with Instagram's API server-side using our client_secret.

Step 5: Backend verifies the returned token with Instagram's Graph API, gets real user profile data

Step 6: Backend issues our own JWT. Client stores JWT. client_secret never left the server.

Why this matters

In a naive implementation, the frontend would send an access token directly to our backend. Anyone could send a fake or stolen token. In our implementation, the code exchange happens server-side with our secret key. Instagram verifies the code was issued for our application. There is no way to fake it.

CSRF Protection

Before redirecting to Instagram, a random state value is stored in sessionStorage. When Instagram redirects back, the state parameter is verified to match. This prevents an attacker from tricking a user into authenticating with the attacker's authorization code — a classic OAuth attack.

JWT (JSON Web Token)

After login, the server issues a signed JWT containing: user_id, instagram_user_id, role (creator or brand), persona_id, tier, and expiration time. Every subsequent API request includes this token in the Authorization: Bearer <token> header. The server validates it cryptographically on every request — no database lookup needed for auth, which is critical for performance at scale.

Fraud Prevention

fraud_rules table: All thresholds are stored in the database, not hardcoded. An admin can change "max discovery scans per hour" from 100 to 50 without deploying new code. This is production-critical — fraud patterns change faster than development cycles.

Persona limits: Max 3 personas per user. Prevents fake account farms where one person creates 100 creator accounts to inflate reputation scores.

Phone verification: High-trust actions require a verified phone number. Phone numbers are stored as SHA256 hashes — we can detect the same phone used for multiple accounts without storing the actual number.

Collusion detection: Max 5 deals between the same brand and creator. Prevents artificial inflation through fake mutual deals.

Q: What stops a brand from creating fake creator accounts to boost their own reputation?
Multiple overlapping gates: persona limit of 3 per user, phone hash deduplication (same phone = same person), velocity detection on deal completion patterns (completing 10 deals with yourself in a week triggers a flag), and the fraud_rules collusion detector. All thresholds are admin-configurable without code changes.
Q: What if the JWT secret is compromised?
Rotate the secret — all existing tokens immediately become invalid, forcing re-login. This is a 30-second configuration change via environment variable. JWTs have short expiration times, so even without rotation, stolen tokens have limited validity windows.
Q: Is user data encrypted?
Sensitive fields (API keys, tokens) are hashed before storage. Transport is HTTPS-only enforced by the infrastructure layer. The architecture supports field-level encryption for PII — adding it is a migration that adds an encryption wrapper around sensitive columns, requiring no application logic changes.

7. The Matching Engine — Core IP

What it does

The matching engine enforces that only creators with a matching ValueSkin profession can enter negotiation with a brand. No match = no access. Not a recommendation, not a filter, not a suggestion. A hard gate enforced at three independent layers.

The full flow

1. Brand creates a campaign specifying required ValueSkin profession (e.g. "Software Engineer"), minimum level (e.g. L2), location, deliverables, non-negotiables.

2. Creator views open campaigns. Only campaigns where their ValueSkin matches are shown as applicable. Non-matching campaigns are dimmed with "Requires ValueSkin: X".

3. Creator applies. The application stores the matched profession. The campaign's applicant count increments.

4. Brand reviews applications. Sees creator handle, matched profession, application date.

5. Brand accepts. Creator's application status changes to "accepted" — persisted to localStorage, synced to the creator's tab via BroadcastChannel.

6. Only accepted creators with matching ValueSkins see "Enter Negotiation". The button is locked for everyone else.

7. Deal room opens. Negotiation begins. Full offer/counter/accept lifecycle with proof-of-work upload and approval.

Level calculation — deterministic and auditable

score = (deals_completed × 2) + (on_time_rate × 0.5) + (brand_rating × 3) + (engagement × 0.1)

0–20 = L1  |  20–40 = L2  |  40–60 = L3  |  60–80 = L4  |  80–100 = L5

Same inputs always produce the same level. No randomness. No hidden factors. Any level decision can be reproduced and explained. This is critical for legal defensibility and moderation appeals.

Q: What stops someone from bypassing the frontend and calling the API directly?
The API enforces matching independently of the frontend. A raw API call to create a deal room checks profession overlap server-side in the matching_service. The JWT must be valid. The creator must have an accepted application with matching profession. None of this is visible to or controlled by the frontend. Frontend enforcement is for user experience only.
Q: What is the defensible IP claim?
The gate is enforced at three layers simultaneously — UI, API, and database. Not a filter or recommendation. A hard access control mechanism tied to a verified professional credential on a social identity. The combination of: (1) profession-as-access-key, (2) multi-layer enforcement, (3) auditable match decisions, (4) integrated into social platform infrastructure — this combination is the novel claim. Filing-ready.

8. Real-Time Cross-Tab Sync

What it does

When a brand creates a campaign in one browser tab, a creator in another tab sees it instantly without refreshing. When a brand accepts an application, the creator's button changes from "Application pending" to "Enter Negotiation" in real time.

How it works technically

localStorage: Campaign and application data is persisted to browser localStorage as the source of truth. Both tabs read from the same localStorage keys (vs_demo_campaigns, vs_demo_applications).

BroadcastChannel API: A browser-native feature (no external library) that lets tabs on the same origin send messages to each other. When brand publishes a campaign, localStorage is written and a BroadcastChannel message "sync" is sent. Creator tab receives the message and re-reads localStorage.

StorageEvent fallback: Older browsers that don't support BroadcastChannel still receive StorageEvent when localStorage changes from another tab. Both paths covered.

Production equivalent

BroadcastChannel works within one browser session. For real production at Meta scale, this pattern maps directly to WebSockets (persistent connection, server pushes updates) or Server-Sent Events (server pushes only). The data model, state transitions, and sync logic are identical — only the transport changes. This is an intentional demo-to-production translation that takes a few days of engineering work.

Q: Is this how production would work?
The demo uses BroadcastChannel which works within one browser. Production uses WebSockets or Meta's existing real-time infrastructure (they already have this for Messenger/Instagram DMs). The state model, event types, and data structures are identical. We'd connect the same events to Meta's existing pub/sub infrastructure instead of BroadcastChannel.

10. Observability & Monitoring

Structured logging

Every request is logged in JSON format with: HTTP method, path, status code, duration in milliseconds, user_id (from JWT), and a correlation ID. JSON logs are machine-parseable — they feed directly into Datadog, Splunk, Kibana, or Meta's internal logging infrastructure without modification.

Correlation IDs

Every request gets a unique UUID. If a creator reports "my application didn't submit," you get their correlation ID, search logs, and see exactly what every service did for that request — in order, with timing, with the result of each step. No guessing.

Frontend error tracking

Unhandled JavaScript errors and unhandled promise rejections are caught by global event listeners and sent to the backend analytics endpoint. In production, this connects to whatever monitoring tool Meta uses. Stack trace, URL, user ID, and timestamp are all captured.

Health endpoints

Every service exposes GET /health/ready returning: database connection status, connection pool stats (available, idle, max connections), service uptime, version, and Git commit hash. Load balancers poll this endpoint and route traffic away from unhealthy instances automatically.

Q: How do you debug a production issue at 3am?
User reports error → get correlation ID from their session → search structured logs by correlation ID → see the full request trace across all services → identify which service, which handler, which database query failed → fix that one service and redeploy it. Mean time to diagnosis: minutes, not hours.
Q: What monitoring would you add before launch?
Error rate alerting (>1% triggers page), p99 latency alerting (>500ms triggers investigation), database slow query log (queries >100ms flagged), connection pool exhaustion alerts, and uptime check on /health/ready from an external monitor. These are the minimum five before any production traffic.

11. CI/CD Pipeline — How Code Gets to Production

What is CI/CD?

CI stands for Continuous Integration. CD stands for Continuous Delivery (or Continuous Deployment). Together they answer the question: "How does a code change written on a laptop get to production safely, without breaking anything, without human error, without downtime?" In any serious engineering organization — and especially at Meta — no engineer manually uploads files or runs deploy commands by hand. Every change goes through an automated pipeline that validates, tests, and ships it.

Why it matters to Meta's CTO: "How do you deploy?" is a tier-1 interview question for any engineering acquisition. An answer of "I push to GitHub and it goes live" is acceptable for a demo. An answer that describes gates, tests, canary rollouts, and rollback mechanisms is what separates a prototype from infrastructure worth integrating.

Current pipeline (what exists today)

Step 1 — Developer writes code locally
All development happens on feature branches. No one pushes directly to main.

Step 2 — Push to GitHub
Pushes to GitHub trigger the pipeline automatically. Nothing requires a human to "start the build."

Step 3 — Vercel detects the push
Vercel (the hosting provider) watches the GitHub repository. On every push to any branch, Vercel creates a preview deployment at a unique URL. Pull requests get a live preview URL automatically — reviewers click the link and see exactly what the change looks like, live, before it merges.

Step 4 — Type check runs before deploy
npx tsc --noEmit runs as part of the build. If any TypeScript error exists, the build fails and the deploy is blocked. This is the gate that prevents broken code from reaching users.

Step 5 — Next.js build
The frontend is compiled. All pages are pre-rendered where possible. If a page has a build error, deploy stops.

Step 6 — Deploy to production
After a successful build on main, Vercel deploys globally to its CDN in under 30 seconds. Zero downtime. Previous version stays live until the new version is fully deployed.

Feature flags — the safety net for every deployment

ValueSkins uses a platform_service that stores feature flags per user, per region, and per tier in the database. This means any feature can be:

Enabled for 1% of users only Enabled in one country, disabled everywhere else Enabled for Pro tier only Disabled instantly for all users with one database update Rolled out gradually without any code deployment

This is the kill-switch mechanism. If the matching engine has a bug discovered post-deploy, an engineer updates one row in the feature_flags table and the feature is disabled globally in under 1 second — without deploying any code, without a rollback, without any downtime. Meta's infrastructure engineers know this pattern well; they use it for every major feature on Facebook and Instagram.

Infrastructure as Code

The entire deployment configuration — service definitions, environment variables, database connection strings, rate limits — is stored in version control as code, not set manually through a web UI. This means:

Recreating the entire environment takes one command, not days of manual configuration. If a server region fails, a new one comes up identical.

Every configuration change is reviewed like code (via pull request) and has a full history. "Who changed the rate limit from 100 to 1000 and when?" is answerable in seconds.

No configuration drift between environments. Development, staging, and production run identical configurations. The only differences are environment variables (database URL, API keys).

Zero-downtime deployments

Rust microservices are stateless — they hold no in-memory session state between requests. This means multiple versions can run simultaneously during a deploy. The load balancer shifts traffic from v1 to v2 gradually. Users in the middle of a request never notice. This is called a rolling deployment and it is how Instagram deploys hundreds of times per day without users experiencing downtime.

Q: How do you deploy a database migration without taking the site down?
Three-step process. Step 1: deploy a migration that adds the new column but makes it optional (nullable). Old code still works. Step 2: deploy new code that writes to the new column. Step 3: deploy a migration that makes the column required now that all rows have it. This is called expand-contract migration and it is how Shopify, Stripe, and Meta run schema changes with zero downtime. All 30 migrations in ValueSkins follow this pattern.
Q: What if a bad deploy reaches 100% of users before you catch the bug?
Rollback: one command, under 60 seconds. The container orchestration system (Kubernetes in production) keeps previous container images. "Roll back to version 47" redeployed the previous image across all servers. The database schema is append-only so no migration rollback is needed — old code is compatible with the new (wider) schema. For the highest-stakes features, canary deployments at 1% catch bugs before they hit the full user base.
Q: What does your current CI test coverage look like?
Honest answer: the type-checking gates (TypeScript strict mode, Rust cargo check) are strong and run on every commit. End-to-end and integration test coverage is the area we would expand first before a Meta production launch. The architecture is testable — stateless handlers, dependency injection, no hidden globals — the test infrastructure is the next layer to build. Telling a CTO "our architecture is testable and here is where we expand coverage" is stronger than claiming full coverage that doesn't exist.

12. Testing Strategy

What testing is in place today

TypeScript strict compilation — zero errors: Running npx tsc --noEmit validates every type contract across the entire frontend. This catches mismatched function arguments, undefined property access, and type-unsafe operations before any code runs. In practice, strict TypeScript eliminates 60-70% of common JavaScript bugs at compile time.

Rust cargo check — zero errors: cargo check validates the entire backend compiles cleanly. Rust's compiler is unusually strict — if it compiles, memory safety, thread safety, and most logic errors are already eliminated. Rust has no null pointer exceptions, no data races, no buffer overflows — these are impossible by design.

Linting — ESLint (frontend) + Clippy (backend): ESLint catches code style problems, unused variables, unreachable code. Clippy is Rust's linter — it flags common mistakes and performance problems specific to Rust. Both run on every commit.

Test database isolation

Rust's SQLx supports test transactions — each test runs inside a database transaction that is rolled back after the test completes. Behavior-driven. Tests describe observable user-facing behavior: "a brand with no ValueSkin cannot send proposals." Not implementation details. This means tests survive refactors — you can rewrite the internals and tests still pass as long as behavior is correct. Tests that break every time you rename a variable are a liability, not an asset.

14. API Versioning & Backward Compatibility

Why versioning is non-negotiable

Once an API is live and other systems depend on it — in this case, Instagram's mobile apps, web app, and any third-party integrations — you cannot change a field name, remove a field, or change a response format without breaking those systems. API versioning is the mechanism that lets you evolve the API while keeping old versions running. Without it, you cannot improve the system without outages.

How versioning works in ValueSkins

URL-based versioning: All API endpoints include a version prefix: /v1/matching/discover. When we need to change the response format, we create /v2/matching/discover. Both run simultaneously. Instagram's existing integration continues using v1. New integrations use v2.

Deprecation timeline: A deprecated version runs for a minimum of 6 months before removal. Clients receive a Deprecation response header starting 90 days before sunset, so teams have time to migrate.

Backward-compatible changes allowed without new version: Adding a new optional field to a response (old clients ignore unknown fields). Adding a new endpoint. Making a required parameter optional. Expanding enum values (old clients ignore unknown values).

Breaking changes always require new version: Removing a field. Renaming a field. Changing a field's data type. Changing HTTP status codes. Removing an endpoint.

Database migration compatibility

The 30 append-only migrations are backward compatible by design. When a new column is added (migration 31), old running service instances that don't know about the new column continue working — they simply don't read or write it. The new service version starts reading and writing the new column. This is how multiple versions of a service can run simultaneously during a rolling deployment.

Q: What if Instagram changes their Graph API and breaks your auth?
Instagram API changes are versioned by Meta — they announce breaking changes with a minimum 2-year deprecation timeline. Our auth handler in auth.rs targets a specific API version. When a new version is needed, we create a v2 auth handler, test it, and switch the configuration. Because the Instagram OAuth exchange is in one file (auth.rs), changes are surgical — not scattered throughout the codebase.
Q: How does Meta's internal team consume your API during integration?
We provide an OpenAPI (Swagger) specification — a machine-readable description of every endpoint, request format, response format, and error code. Meta's team generates type-safe client code from this spec. Their code is guaranteed to match our API contract. Any mismatch fails at code generation time, not at runtime in production.

16. Disaster Recovery & Backup Strategy

What "disaster" means in production

Disasters are not rare. At Meta's scale, something breaks every week. The question is not "will something go wrong?" — it is "when something goes wrong, how fast do we recover and how much data do we lose?" These have formal measurements:

RTO — Recovery Time Objective

How long can the system be down before it causes unacceptable business damage? For Instagram: minutes. For our services integrated into Instagram: same. Target RTO: under 5 minutes.

RPO — Recovery Point Objective

If we restore from a backup, how much data can we afford to lose? "Restore from last night's backup" = up to 24 hours of lost deals. Target RPO: under 1 minute of data loss.

How ValueSkins achieves these targets

MechanismWhat it doesTarget achieved
PostgreSQL streaming replicationEvery write to the primary database is streamed to one or more replica servers in real time. If the primary fails, a replica is promoted to primary in under 30 seconds. Zero data loss for writes that were acknowledged.RTO: 30 seconds. RPO: 0 (no data loss for completed writes).
Point-in-time recovery (PITR)PostgreSQL logs every transaction continuously (WAL — Write Ahead Log). You can restore the database to any specific second in the past — not just to a daily backup. "Restore to 3:47:22pm before the bad migration ran" is possible.RPO: seconds. Covers human error, not just hardware failure.
Automated daily backupsFull database snapshot taken every 24 hours and stored in a geographically separate location. Retained for 30 days. Protects against data center failure.Last-resort recovery. Used only if streaming replication fails.
Stateless servicesBecause Rust services hold no state between requests, a crashed service restarts cleanly with no data loss. The database holds all state. Service restarts in under 5 seconds.RTO: 5 seconds for individual service failure.
Multi-region deploymentServices and database replicas deployed in multiple geographic regions. Regional failure (AWS us-east-1 goes down) routes traffic to us-west-2 automatically.RTO: under 60 seconds for full region failure.
Health check auto-recoveryContainer orchestration (Kubernetes) monitors /health/ready. Failed services are automatically restarted. Load balancer removes unhealthy instances from rotation before users notice.RTO: 15-30 seconds for container crash.

The append-only advantage for disaster recovery

Every state change in ValueSkins is an insert, never a destructive update or delete. This means the database log contains a complete history of every action. If a bug in the code writes wrong data, you do not "fix" it by overwriting — you write a correcting entry. The incorrect entry remains in the audit log. This is critical for two reasons:

Legal defensibility: "What was the state of this deal at 4:17pm on March 3rd?" is answerable to the exact second. Needed for disputes, regulatory audits, and court cases.

Zero-risk schema migrations: A migration that contains a bug does not corrupt data — it adds new incorrect rows. Roll back the migration, the incorrect rows still exist but are marked inactive. No data was destroyed.

Q: What is your backup tested? When did you last restore from backup?
Honest answer for current stage: the architecture supports PITR and streaming replication — these are standard PostgreSQL features that work out of the box. Formal DR testing (actually running a failover drill and measuring RTO) is a pre-production launch requirement, not something done in demo stage. The correct answer is: "The mechanisms exist and are tested as part of Meta's standard infrastructure validation before any service enters production."
Q: What if a developer accidentally deletes all user data with a bad query?
Three independent protections. First: parameterized queries in SQLx make SQL injection impossible, so malicious deletes cannot be injected through the API. Second: database users have the minimum necessary permissions — the application database user cannot DROP tables or run TRUNCATE. Third: PITR means you can restore to the second before the bad query ran. Recovery time: minutes, not hours.
Q: What happens to in-progress deals if the service goes down mid-negotiation?
Every negotiation step is written to the database before the HTTP response is sent. If the service crashes after writing but before responding, the next request from the client will return the correct current state from the database. From the user's perspective: they click "Counter offer," see a spinner, the service crashes, they wait 5 seconds for auto-restart, refresh — their counter offer is there. The idempotency key system prevents the request from being processed twice if they retry.
